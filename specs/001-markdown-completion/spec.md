# Feature Specification: Markdown Completion Provider

**Feature Branch**: `001-markdown-completion`

**Created**: 2026-05-13

**Status**: Draft

**Input**: User description: "Markdown を書くときに入力補助をする VS Code 拡張機能。独自トリガー文字で補完候補を表示し、SnippetString で記法を挿入する。"

## Clarifications

### Session 2026-05-13

- Q: 候補ラベル（label と documentation）の表示言語をどうするか → A: MVP では英語ラベル固定とし、将来の VS Code 表示言語連動に拡張可能な設計にする
- Q: 拡張のアクティベーション戦略をどうするか → A: onLanguage:markdown と onCommand:Markdown: Insert Syntax の遅延起動を採用する
- Q: 用語を「トリガー文字」と「プレフィックス」のどちらで統一するか → A: 正規用語は「トリガー文字」に統一する
- Q: YAML front matter 内での補完抑制を要件として必須化するか → A: MVP で必須（front matter 内は常に補完抑制）
- Q: MVP の対応 VS Code 最低バージョンをどうするか → A: VS Code 1.85 以上をサポート対象に固定する

## User Scenarios & Testing *(mandatory)*

### User Story 1 - トリガー文字による記法候補の表示と挿入 (Priority: P1)

Markdown ファイルを編集中のユーザーがトリガー文字（`;`）を入力すると、見出し・太字・リンク・コードブロックなど主要な Markdown 記法の候補一覧がポップアップで表示される。候補を選択すると、カーソル位置にタブストップ付きのスニペットとして記法が挿入され、トリガー文字列は消える。

**Why this priority**: 拡張機能のコアバリューそのもの。これだけで最小限の製品として成立し、「記法を覚えなくても候補から選べる」体験を提供する。

**Independent Test**: VS Code で `.md` ファイルを開き、`;` を入力して候補一覧が出ること、候補選択後に正しい Markdown 記法がスニペットとして挿入されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが Markdown ファイルを編集中, **When** `;` を入力する, **Then** Markdown 記法の候補一覧が表示される
2. **Given** 候補一覧が表示されている, **When** 「見出し1」を選択する, **Then** `# ` がカーソル位置に挿入され、`;` は消え、カーソルが見出しテキスト入力位置に移動する
3. **Given** 候補一覧が表示されている, **When** `;link` と入力して絞り込む, **Then** リンク関連の候補だけに絞り込まれる
4. **Given** 「リンク」候補を選択する, **When** 挿入される, **Then** `[text](url)` 形式のスニペットが挿入され、最初のタブストップが `text` 部分にある
5. **Given** ユーザーが Markdown ファイル以外を編集中, **When** `;` を入力する, **Then** Markdown 記法候補は表示されない

---

### User Story 2 - コードブロック内での補完抑制 (Priority: P2)

ユーザーがフェンスドコードブロック（` ``` ` で囲まれた範囲）やインラインコード（`` ` `` で囲まれた範囲）の内部で `;` を入力しても、Markdown 記法の候補は表示されない。コード領域での誤動作を防ぎ、通常のコード入力を妨げない。

**Why this priority**: コードブロック内での誤爆はユーザー体験を大きく損なう。MVP に近い優先度だが、コア機能の後に対応することで段階的に品質を上げられる。

**Independent Test**: フェンスドコードブロック内で `;` を入力し、候補が出ないことを確認する。

**Acceptance Scenarios**:

1. **Given** カーソルがフェンスドコードブロック内にある, **When** `;` を入力する, **Then** Markdown 記法候補は表示されない
2. **Given** カーソルがインラインコード内にある, **When** `;` を入力する, **Then** Markdown 記法候補は表示されない
3. **Given** カーソルがコードブロックの外にある, **When** `;` を入力する, **Then** 通常通り候補が表示される

---

### User Story 3 - コマンドパレットからの記法挿入 (Priority: P3)

ユーザーがコマンドパレットまたは右クリックメニューから「Markdown: Insert Syntax」コマンドを実行すると、Quick Pick で記法一覧が表示され、選択した記法がカーソル位置に挿入される。補完ポップアップが出ない場面でも記法を挿入できるフォールバック手段を提供する。

**Why this priority**: 補完トリガーに気づかないユーザーや、補完が動作しない設定環境でも機能を使えるようにする補助的な導線。コア機能ではないが発見可能性を高める。

**Independent Test**: コマンドパレットで「Markdown: Insert Syntax」を実行し、Quick Pick から記法を選択して挿入されることを確認する。

**Acceptance Scenarios**:

1. **Given** Markdown ファイルを編集中, **When** コマンドパレットから「Markdown: Insert Syntax」を実行する, **Then** 記法一覧が Quick Pick として表示される
2. **Given** Quick Pick で「表 (3列)」を選択する, **When** 実行する, **Then** 3列のテーブルテンプレートがスニペットとして挿入される

---

### Edge Cases

- ユーザーが `;` を含む通常テキスト（例: 英語のセミコロン利用）を入力した場合、候補は表示されるが ESC で閉じれば通常入力が継続される
- 空のファイルで `;` を入力した場合でも候補が正常に表示される
- 非常に長い行の末尾で `;` を入力した場合でも補完が正常に動作する
- YAML front matter（`---` で囲まれた先頭ブロック）内では候補を抑制する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 拡張機能は Markdown ファイル（言語識別子 `markdown`）に対してのみ補完候補を提供しなければならない
- **FR-002**: トリガー文字 `;` の入力時に補完候補一覧を表示しなければならない
- **FR-003**: 補完候補として少なくとも以下の Markdown 記法を提供しなければならない: 見出し（H1〜H3）、太字、斜体、リンク、画像、表、コードブロック、引用、脚注、水平線
- **FR-004**: 各補完候補はタブストップ付きスニペットとして挿入し、ユーザーが Tab キーで可変部分を移動できなければならない
- **FR-005**: トリガー文字以降の入力（例: `;link`）で候補を絞り込めなければならない
- **FR-006**: 候補選択時にトリガー文字列（`;link` 等）を置換して記法本体に差し替えなければならない
- **FR-007**: 候補の `label` は MVP では英語で表示し（例: "Link", "Table (3 cols)"）、`documentation` に記法例を短く表示しなければならない
- **FR-011**: 候補表示文言は将来の表示言語連動に備えてローカライズ可能な構造で管理しなければならない（MVP 時点では英語リソースのみ提供）
- **FR-008**: フェンスドコードブロック内およびインラインコード内では補完候補を抑制しなければならない
- **FR-009**: コマンドパレットから呼び出せる「Markdown: Insert Syntax」コマンドを提供し、Quick Pick で記法を選択・挿入できなければならない
- **FR-010**: 補完候補の並び順は使用頻度の高い記法が上位に来るよう `sortText` で制御しなければならない
- **FR-012**: 拡張の activationEvents は MVP で `onLanguage:markdown` と `onCommand:Markdown: Insert Syntax` を採用し、不要な常時起動を避けなければならない
- **FR-013**: 仕様上の正規用語は「トリガー文字」に統一し、`;link` のような入力は「トリガー文字から始まる入力列」として扱わなければならない
- **FR-014**: YAML front matter（`--- ... ---`）内では補完候補を常に抑制しなければならない
- **FR-015**: MVP の対応対象は VS Code 1.85 以上とし、それ未満のバージョンはサポート対象外としなければならない

### Security & Communication Constraints *(mandatory)*

- **SCON-001**: 本機能は外部通信パスを一切持たない。ネットワークアクセスは不要であり、すべての処理はローカルで完結する
- **SCON-002**: 本機能はテレメトリ、更新チェック、外部 API 呼び出しを行わない。ネットワーク接続は使用しない
- **SCON-003**: 外部通信がないため、ネットワーク未接続シナリオの特別な処理は不要。完全オフライン動作する
- **SCON-004**: 最小必須テストとして、(1) トリガー文字入力時に正しい補完候補が返されるプライマリパステスト、(2) コードブロック内で補完が抑制される境界条件テストを実施する

### Key Entities

- **CompletionSnippet**: 1つの Markdown 記法候補を表す。属性: 識別キー、表示ラベル、説明文、スニペット文字列、並び順、フィルターテキスト
- **DocumentContext**: 補完要求時のカーソル位置の文脈情報。属性: コードブロック内か否か、行頭か否か、現在行のトリガー文字列

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがトリガー文字入力から候補選択・記法挿入まで 3 秒以内に完了できる
- **SC-002**: 提供する全候補（10 種以上）が正しいスニペットとして挿入でき、タブストップが機能する
- **SC-003**: コードブロック内でのトリガー入力時に候補が表示されない（誤爆率 0%）
- **SC-004**: コマンドパレット経由でも全記法が挿入可能である
- **SC-005**: 拡張機能がネットワーク通信を一切行わないことを検証できる

## Assumptions

- ユーザーは VS Code で Markdown ファイルを日常的に編集している
- トリガー文字 `;` は Markdown 本文で頻繁に使用される文字ではなく、誤爆は許容範囲に収まる
- MVP ではトリガー文字の変更機能（ユーザー設定）はスコープ外とする
- MVP では選択テキストの変換（選択中に `;link` でリンク化など）はスコープ外とする
- MVP では GFM / Zenn / Qiita 向けのテンプレート切替はスコープ外とする
- MVP では候補ラベルの実表示言語は英語固定とし、表示言語連動は将来拡張とする
- 対象は標準的な CommonMark + 一般的な拡張記法（脚注、表）とし、特殊な方言への対応は v1 では行わない
- VS Code の `editor.quickSuggestions` 設定に関わらず、トリガー文字による補完は動作する（VS Code API の仕様による）
- MVP の対応 VS Code バージョンは 1.85 以上とする
