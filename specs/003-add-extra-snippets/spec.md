# Feature Specification: Additional Markdown Snippets

**Feature Branch**: `003-add-extra-snippets`

**Created**: 2026-05-21

**Status**: Draft

**Input**: User description: "GitHub issue #4『Snippetの追加』をもとに、下線と折りたたみの snippet を追加する。追加候補は `<u>text</u>` と `<details><summary>summary</summary>body</details>`。"

## Clarifications

### Session 2026-05-21

- Q: SC-001 の「既存の記法候補操作と同じ手順数」をどう測定するか → A: 候補一覧表示後、2 操作以内で挿入完了できる

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 下線をすぐ挿入できる (Priority: P1)

Markdown を編集中のユーザーが、既存の記法候補と同じ導線から下線記法を選び、カーソル位置にそのまま挿入できる。

**Why this priority**: issue で明示された追加要望のうち最小の 1 機能であり、既存の記法補助を崩さずに編集効率を直接上げる。

**Independent Test**: Markdown 編集中に記法候補一覧またはコマンド経由の選択一覧から下線を選び、`<u>text</u>` 形式で挿入されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが Markdown 文書を編集中, **When** 記法候補一覧から下線を選択する, **Then** `<u>text</u>` 形式のスニペットがカーソル位置に挿入される
2. **Given** ユーザーが記法選択コマンドを開いている, **When** 下線を選択する, **Then** 同じ下線スニペットが挿入される
3. **Given** 下線スニペットが挿入された直後, **When** ユーザーが入力を続ける, **Then** 最初の編集位置は `text` 部分になる

---

### User Story 2 - 折りたたみブロックをテンプレート付きで挿入できる (Priority: P2)

Markdown を編集中のユーザーが、詳細説明や補足を折りたたみ表示したいときに、summary と body を含む複数行テンプレートを一度で挿入できる。

**Why this priority**: 折りたたみは issue で明示された追加要望であり、複数行テンプレートを手入力する負担が大きいため自動挿入の価値が高い。

**Independent Test**: Markdown 編集中に折りたたみ候補を選択し、`<details>`、`<summary>summary</summary>`、`body`、`</details>` を含む複数行テンプレートが実改行で挿入されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが Markdown 文書を編集中, **When** 記法候補一覧から折りたたみを選択する, **Then** 複数行の折りたたみテンプレートがカーソル位置に挿入される
2. **Given** 折りたたみテンプレートが挿入された直後, **When** ユーザーが Tab で編集位置を移動する, **Then** `summary` と `body` を順に編集できる
3. **Given** 折りたたみ候補をコマンド経由で選択する, **When** 挿入する, **Then** 実改行を含む同一テンプレートが挿入され、文字列 `\\n` は本文に出力されない

---

### User Story 3 - 新しい snippet が既存一覧に自然に統合される (Priority: P3)

既存ユーザーが新しい記法を探す際に、候補一覧とコマンド一覧のどちらでも追加された項目を見つけられ、既存項目と同じ粒度の表示情報で判断できる。

**Why this priority**: 単に挿入できるだけでは発見しづらく、一覧品質が崩れると既存機能全体の使い勝手を損なうため。

**Independent Test**: 候補一覧とコマンド一覧を表示し、新しい 2 項目が既存候補と同様に名称・簡易説明付きで表示され、対象文字列で絞り込めることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが候補一覧を開いている, **When** 新しい項目を確認する, **Then** 下線と折りたたみが既存候補と同じ表示粒度で並ぶ
2. **Given** ユーザーが絞り込み入力を行う, **When** 下線または折りたたみに対応する語で検索する, **Then** 対応する候補へ到達できる
3. **Given** コマンド経由の一覧を開いている, **When** 新しい項目を確認する, **Then** 候補一覧と同じ 2 項目が選択可能である

### Edge Cases

- 下線と折りたたみのどちらを追加しても、既存の記法候補数や既存候補の挙動を壊さないこと
- 折りたたみテンプレートは空の Markdown 文書先頭でも正しいインデントと改行で挿入されること
- HTML タグを利用する追加候補でも、Markdown 以外の文書では新規候補を表示しないこと
- 候補一覧とコマンド一覧で片方にだけ新規項目が存在する不整合を発生させないこと

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは Markdown 編集中の記法候補一覧に、下線と折りたたみの 2 項目を追加しなければならない
- **FR-002**: システムは下線候補を選択したとき、`<u>text</u>` 形式のスニペットを挿入し、最初の編集位置を `text` 部分に置かなければならない
- **FR-003**: システムは折りたたみ候補を選択したとき、`<details>`、`<summary>summary</summary>`、`body`、`</details>` から成るテンプレートを挿入し、`summary` と `body` を順に編集できなければならない
- **FR-004**: システムはコマンド経由の記法選択一覧でも、下線と折りたたみの 2 項目を選択可能にしなければならない
- **FR-005**: システムは候補一覧とコマンド一覧で、各新規項目の名称と簡易説明を既存候補と同等の粒度で表示しなければならない
- **FR-006**: システムは下線と折りたたみの両候補を、既存の絞り込み入力から到達可能にしなければならない
- **FR-007**: システムは折りたたみテンプレートを実改行で挿入しなければならず、文字列 `\\n` を本文へ出力してはならない
- **FR-008**: システムは新規 2 項目を既存候補一覧へ追加しても、既存候補の表示順と既存項目の挿入内容を変えてはならない
- **FR-009**: システムは Markdown 以外の文書では、新規 2 項目を含む記法候補を表示してはならない
- **FR-010**: システムは自動テストで、下線の挿入内容、折りたたみの複数行挿入内容、候補一覧とコマンド一覧への反映を検証しなければならない

### Security & Communication Constraints *(mandatory)*

- **SCON-001**: 本機能は外部通信パスを追加してはならず、入力補助はローカル処理のみで完結しなければならない
- **SCON-002**: 本機能はテレメトリ、解析送信、外部サービス照会を行ってはならない
- **SCON-003**: ネットワーク未接続時でも新しい 2 項目の表示と挿入は完全に動作しなければならない
- **SCON-004**: 最小必須自動テストとして、1 つの主要経路（新規候補の正常挿入）と 1 つの境界条件（折りたたみの複数行挿入で `\\n` を出力しないこと）を含めなければならない

### Key Entities

- **Snippet Entry**: 記法候補 1 件を表す。属性は表示名、簡易説明、挿入テンプレート、絞り込み用文字列、一覧内での並び位置
- **Insertion Surface**: 記法を選択できる導線を表す。候補一覧とコマンド経由一覧の 2 種類があり、どちらでも同一内容を提供する

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは下線または折りたたみを、候補一覧表示後 2 操作以内で挿入完了できる
- **SC-002**: 新しい 2 項目は候補一覧とコマンド一覧の両方で 100% 表示される
- **SC-003**: 折りたたみテンプレートは 100% の検証で実改行として挿入され、文字列 `\\n` を出力しない
- **SC-004**: 既存候補の挿入内容と一覧順序に回帰が発生しないことを自動テストで確認できる

## Assumptions

- 既存の記法追加方針に従い、新規候補も既存候補と同じ導線から提供する
- 下線は HTML ベースの Markdown 記法として扱い、追加の変換やレンダリング保証は本機能の範囲外とする
- 折りたたみテンプレートの初期内容は issue 記載の `summary` と `body` をそのまま初期値として採用する
- 候補名称は既存の英語ラベル方針に合わせる
